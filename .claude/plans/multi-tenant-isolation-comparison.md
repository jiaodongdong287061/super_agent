# 企业级多租户数据隔离方案对比：分集合 vs Metadata 过滤

> 适用场景：RAG 知识库中不同部门/租户的数据隔离

## 1. 方案概述

### 分集合（Collection Isolation）
每个部门/租户拥有独立的向量集合（Collection），数据物理分离。

```
qdrant/
├── super_agent_docs_103    ← 部门 103 的文档
├── super_agent_docs_201    ← 部门 201 的文档
├── super_agent_docs_207    ← 部门 207 的文档
└── super_agent_docs        ← 公开/无部门文档
```

检索流程：
- 普通用户：`get_store(tenant_id=103)` → 直接查对应集合
- 超管：`FanOutRetriever` → 查所有集合 → RRF 融合结果

### Metadata 过滤（Metadata Filtering）
所有数据存在同一集合，通过 metadata 字段区分所属部门。

```
一个集合 super_agent_docs
├── chunk_A  → metadata = { department: "103", allowed_roles: ["admin"] }
├── chunk_B  → metadata = { department: "103", allowed_roles: ["ops"] }
├── chunk_C  → metadata = { department: "201", allowed_roles: ["admin"] }
└── chunk_D  → metadata = { department: "" }  ← 公开文档
```

检索流程：
- 普通用户：`search(query, filter={department: {"$eq": "103"}})` → 一次查询
- 超管：去掉 department 过滤条件 → 查全量 → 一次查询

## 2. 全方位对比

### 2.1 数据安全与合规

| 维度 | 分集合 | Metadata 过滤 |
|------|--------|---------------|
| **隔离级别** | 物理隔离。存储层即分离，应用层穿透不了 | 逻辑隔离。仅依赖应用层 filter 代码的正确性 |
| **审计认可度** | 高。能向审计证明"部门 A 的数据物理上不可能出现在部门 B 的集合里" | 低。审计会质疑"如果应用有 bug 或直连数据库，就能看到其他部门数据" |
| **等保/GDPR 合规** | 满足。按部门删除数据 = drop collection，无残留 | 有风险。按 filter 删除需要扫描全量数据，可能遗漏 |
| **数据泄露风险源** | 只有一个：应用层 token 校验失效 | 两个：token 校验失效 + filter 逻辑 bug |

### 2.2 运维管理

| 维度 | 分集合 | Metadata 过滤 |
|------|--------|---------------|
| **日常索引** | 50 个部门要调 50 次索引接口，或自动化脚本遍历 | 一个集合，一个索引命令写完 |
| **新部门接入** | 检查集合是否存在 → 不存在则创建（需配向量维度/距离算法）→ 开始索引 | 直接索引，department 字段写进去即可 |
| **部门数据删除** | `drop collection`，瞬间完成 | `delete(where={department: "103"})`，需要扫描全集合匹配 |
| **数据量统计** | 查 N 个集合的 count 再汇总 | 一个 count 加 filter 搞定 |
| **备份恢复** | 按部门独立备份/按部门独立恢复 | 全量备份；单部门恢复需 filter 导出再导入 |
| **故障爆炸半径** | 一个集合损坏/异常只影响对应部门 | 集合损坏则全公司不可用 |
| **版本升级** | 需确保所有集合配置一致，升级需批量操作 | 只需维护一个集合 |

### 2.3 业务灵活性

| 维度 | 分集合 | Metadata 过滤 |
|------|--------|---------------|
| **超管跨部门查询** | 查 N 个集合 → RRF 融合 → 去重排序，多路查询性能开销大 | 不去掉 filter 即可，一次查询 |
| **部门合并**（两个部门变一个） | 将集合 A 的数据搬运到集合 B，复杂 | 批量更新 metadata: `update(department: "A" → "B")` |
| **部门拆分**（一个部门变两个） | 新建集合 B，从集合 A 筛选并复制数据 | 批量更新 metadata，更新部分文档的 department 字段 |
| **文档跨部门共享** | 文档只能属于一个部门，共享需要复制 | 一个文档可标记多个部门或设为公开 |
| **多级权限**（部门内再分角色） | 需要额外的 metadata 字段配合 | 天然支持，同一个 filter 机制即可 |

### 2.4 性能

| 维度 | 分集合 | Metadata 过滤 |
|------|--------|---------------|
| **单部门查询** | 集合小，搜索快 | 集合大但有 filter 索引，Qdrant/Milvus filter 性能优秀 |
| **超管跨部门查询** | N 次搜索 + 结果合并，线性增长 | 1 次搜索（去掉 filter），常数级 |
| **索引写入** | N 个集合各写各的 | 一个集合写入，吞吐高 |
| **大数据量时**（千万级+） | 可按部门拆分到不同节点，天然分片 | filter 过滤在大基数时可能出现性能拐点 |

### 2.5 实现复杂度

| 维度 | 分集合 | Metadata 过滤 |
|------|--------|---------------|
| **代码改动量** | 中等。需要 FanOut 检索器、集合生命周期管理 | 小。已有 metadata 字段，补上写入逻辑即可 |
| **检索器逻辑** | 需区分单部门检索和 FanOut 检索 | 统一检索接口，仅 filter 参数不同 |
| **异常处理** | 新部门首次索引时集合不存在、集合配置不一致 | 几乎没有额外异常 |

## 3. 决策矩阵

| 决策因素 | 权重 | 分集合 | Metadata | 说明 |
|----------|------|--------|----------|------|
| 合规审计要求 | 高 | ✅ 优 | ⚠️ 中 | 有合规审计选分集合 |
| 部门数量 | 相关 | 部门<20 选分集合 | 部门>20 选 metadata | 部门越多，分集合运维成本线性增长 |
| 跨部门查询频率 | 相关 | 低频率选 | 高频率选 | 超管经常跨部门查，metadata 更优 |
| 组织调整频率 | 相关 | 低频选 | 高频选 | 部门经常合并拆分，metadata 灵活 |
| 运维团队规模 | 相关 | 有专职 DBA 选 | 团队小选 | 分集合运维成本更高 |
| 数据敏感度 | 高 | 机密数据选 | 内部资料选 | 涉密数据建议物理隔离 |

### 综合建议

| 场景 | 推荐方案 |
|------|---------|
| 金融/医疗/政务（有等保或合规要求） | **分集合** |
| 内部 IT 知识库（runbook、运维文档） | **Metadata** |
| 两者都有（部分数据涉密，部分公开） | **混合**：涉密部门分集合，普通部门 metadata |
| 超小规模（<5 部门，<10 万文档） | **Metadata**（简单为主） |
| 大规模（>50 部门，文档量亿级） | **分集合**（便于物理分片） |

## 4. 混合方案参考

并非二选一，可以同时使用：

```
集合 super_agent_docs_finance   ← 财务部（高敏感，物理隔离）
集合 super_agent_docs_hr        ← HR 部（高敏感，物理隔离）
集合 super_agent_docs           ← 其他普通部门（metadata 过滤）
    文档 A → department: "103"
    文档 B → department: "201"
```

检索器根据用户部门判断走哪个方案，对上层 API 透明。

## 5. 总结

```
┌─────────────────────────────────────────────────┐
│               企业级数据隔离方案                  │
├─────────────────────────────────────────────────┤
│                                                 │
│  需要合规审计？                                   │
│  ├── 是 → 分集合（物理隔离，审计友好）              │
│  └── 否 → 考虑以下因素：                          │
│         ├── 部门 < 20 且不常变动 → 分集合          │
│         ├── 经常跨部门查询/组织调整 → Metadata      │
│         ├── 运维团队小 → Metadata                  │
│         └── 人员流动大/数据敏感 → 分集合            │
│                                                 │
│  都不完全满足 → 混合方案                           │
│                                                 │
└─────────────────────────────────────────────────┘
```
